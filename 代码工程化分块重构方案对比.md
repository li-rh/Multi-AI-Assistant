# 多AI同步提问助手：单文件重构方案对比（3选1）

## 0. 前提与目标

- 重构约束：**必须保持单文件**（`多AI同步提问助手.js`），不拆多文件、不引入打包构建。
- 当前基础：文件内已存在 9 个分区（`state/config/elements/utils/ui/events/lifecycle/comms/main`）。
- 重构目标：提升可读性、功能分区清晰度、修改时的回归可控性。

---

## 1) 方案A：**注释分层 + 命名规范化（轻量快改）**

> 核心思路：不改变整体架构，只通过“统一命名 + 段落分层 + 入口收敛”把单文件读写成本降下来。

### 结构建议（仍是一个对象）

```text
AITabSync
  ├─ state/config/elements
  ├─ utils
  ├─ ui
  ├─ events
  ├─ lifecycle
  ├─ comms
  └─ main
```

### 具体改造点

1. 给每个模块加统一段头：`[职责] [输入] [输出] [副作用]`。
2. 统一方法命名前缀：
   - `get*/is*/can*` 纯读取
   - `build*/normalize*` 纯转换
   - `apply*/dispatch*/handle*` 副作用
3. 将“常量散点”集中到 `config`，禁用硬编码时间常量。
4. 在每个大模块增加 `public API` 区块，限制外部调用入口。

### 优点

- 成本最低，1~2 天即可完成第一轮。
- 行为变化最小，适合当前线上稳定优先。
- 对现有代码侵入最小，便于快速落地。

### 缺点

- 主要是“可读性治理”，架构性问题缓解有限。
- 长期维护收益一般，复杂逻辑依然混在一个对象里。

---

## 2) 方案B：**单文件伪模块 IIFE 化（推荐）**

> 核心思路：仍在一个文件内，把每个功能域改造成“私有闭包模块 + 显式导出接口”，减少跨域随意访问。

### 结构建议（单文件内多模块）

```text
(function(){
  const Core = (() => { ... return { state, config }; })();
  const Infra = ((core) => { ... return { storage, logger, dom }; })(Core);
  const QuerySync = ((core, infra) => { ... return { init, broadcast, receive }; })(Core, Infra);
  const AssetSync = ((core, infra) => { ... return { init, broadcast, receive }; })(Core, Infra);
  const UI = ((core, infra) => { ... return { init, render }; })(Core, Infra);
  const App = ((...) => { ... return { init }; })(...);
  App.init();
})();
```

### 具体改造点

1. 将 `comms` 拆为 `QuerySync` 与 `AssetSync` 两个 IIFE 模块。
2. 把 `GM_getValue/GM_setValue` 封装到 `Infra.storage`，业务模块禁止直接调用 GM API。
3. 通过“依赖注入”传 `core/infra`，替代模块间直接引用 `AITabSync.*`。
4. 给每个模块定义最小导出面（例如仅导出 `init/handle*`）。

### 优点

- 在单文件约束下，边界清晰度和可维护性提升最大。
- 减少隐式共享状态，回归范围更可控。
- 后续若改多文件，迁移成本最低（模块边界已经就位）。

### 缺点

- 改造量中等，需系统梳理调用关系。
- 初期会出现“接口搬运”工作量。

---

## 3) 方案C：**单文件事件总线 + 流水线编排（适合复杂链路）**

> 核心思路：把文本/资产同步流程改为“事件驱动流水线”，让复杂 if-else 变成可追踪步骤链。

### 结构建议（单文件内流程化）

```text
Bus.emit('QUERY_FOUND')
  -> validate
  -> routeTargets
  -> persistSharedMessage
  -> receiverConsume
  -> submit

Bus.emit('ASSET_FOUND')
  -> dedup
  -> normalize
  -> persistSharedAsset
  -> receive
  -> inject(paste/file/drop)
  -> verify
```

### 具体改造点

1. 新增轻量 `Bus`（`on/emit/off`）用于模块解耦。
2. 将 `processSharedQuery/processSharedAsset` 拆成阶段函数链。
3. 给关键阶段统一埋点：`stage/start/end/error/duration`。
4. 引入 `PipelineContext`（含 `messageId/siteId/targets`）作为流程载体。

### 优点

- 对复杂同步链路最友好，定位问题效率高。
- 业务流程可视化，后续新增步骤（限流/重试）更容易。

### 缺点

- 心智负担高于方案A/B。
- 若团队不熟悉事件驱动，初期维护门槛更高。

---

## 4) 三种“单文件”方案细节对比

| 维度 | 方案A 注释分层 | 方案B IIFE伪模块（推荐） | 方案C 事件总线流水线 |
|---|---|---|---|
| 单文件约束符合度 | 完全符合 | 完全符合 | 完全符合 |
| 改造成本 | 低 | 中 | 中~高 |
| 可读性提升 | 中 | 高 | 高 |
| 边界清晰度 | 中 | 高 | 很高 |
| 回归风险 | 低 | 中 | 中 |
| 调试效率 | 中 | 高 | 很高（需埋点） |
| 团队上手难度 | 低 | 中 | 中~高 |
| 长期收益 | 中 | 高 | 高 |

---

## 5) 推荐结论（你当前项目）

推荐优先选 **方案B（单文件 IIFE 伪模块化）**：

- 你当前代码已具备分区雏形，B 方案可以“顺着现状演进”，风险和收益最平衡。
- 对 `comms` 这类高复杂区能显著降耦合，又不改变单文件发布习惯。
- 后续若你改主意要多文件化，B 方案几乎可以直接平移。

---

## 6) 如果现在就开工（基于方案B）的最小步骤

1. 先抽 2 个闭包模块：`QuerySync`、`AssetSync`（仅搬移，不改行为）。
2. 再抽 `Infra.storage/logger/dom`，统一接管外部 API 与日志。
3. 最后抽 `App` 装配层，统一 `init` 顺序与监听注册。

> 以上步骤全部可在一个文件内完成，并可分 2~3 次提交逐步验证。
